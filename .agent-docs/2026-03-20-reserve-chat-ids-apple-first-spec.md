# Reserve Chat IDs For Apple-First Immediate Thread Navigation

## Goal

Enable immediate navigation for new thread creation on Apple by using real future server `chat.id` values, while preserving the current route/state identity model and allowing basic queued `sendMessage` until chat creation is confirmed by the server.

## Scope

In scope:

- Apple-first flow
- New thread creation only
- `reserveChatIds`
- Durable small Apple reservation pool
- Optimistic local `Chat` and `Dialog` creation using a reserved final `chat.id`
- Immediate navigation to the new thread
- `CreateChatTransaction` using `reservedChatId`
- `SendMessageTransaction` blocked until the chat is confirmed created
- Realtime v2 transaction blockers/prerequisites
- Minimal local pending-chat marker

Out of scope:

- Subthread reservation
- Subthread immediate local navigation
- Local conversation identity refactor
- Full offline-first arbitrary-duration conversation creation
- Media upload dependency implementation
- Cross-client rollout beyond Apple

## Chosen Approach

Use a small durable client-side pool of reserved server chat IDs for new thread creation.

When the user creates a thread:

1. Apple consumes one reserved chat ID from the local pool.
2. Apple creates local optimistic `Chat` and `Dialog` rows using that final future `chat.id`.
3. Apple navigates immediately to `.thread(id: reservedChatId)`.
4. Apple enqueues `CreateChatTransaction(reservedChatId: ...)`.
5. If the user sends messages before create completes, `SendMessageTransaction` is enqueued immediately but blocked on `chatCreated(chatId)`.
6. Once `CreateChatTransaction` succeeds, blocked sends for that chat become runnable.

This preserves the current architecture where route and state identity are based on `chat.id`.

## Server Design

### API

Add:

- `reserveChatIds(count) -> [{ chatId, expiresAt }]`

Extend:

- `createChat(..., reservedChatId?)`

`reservedChatId` is optional. If absent, `createChat` keeps current behavior.

### Reservation Semantics

Reservations are only claims on future `chat.id` values. They are not typed and do not carry payload-specific meaning.

Server guarantees:

- Reserved IDs come from the same underlying sequence used by `chats.id`
- Reservation ownership is bound to `currentUserId`
- Reservations are one-time claimable
- Reservations expire eventually

### Reservation Table

Keep the table minimal:

- `chat_id`
- `user_id`
- `claimed_at`
- `expires_at`
- `created_at`

No token, no type, no extra scope columns.

### `createChat` Claim Path

If `reservedChatId` is present:

1. Verify reservation exists.
2. Verify reservation belongs to the current user.
3. Verify reservation is not claimed.
4. Verify reservation is not expired.
5. Run normal thread creation validation.
6. Insert the real `chats` row using `id = reservedChatId`.
7. Create participants and dialog as normal.
8. Mark reservation claimed in the same transaction.

If `reservedChatId` is absent, keep current behavior unchanged.

### Postgres / Drizzle Change

Current schema uses identity generation for `chats.id`.

For reserved explicit inserts, change `chats.id` to the Postgres shape that allows explicit IDs while still defaulting to sequence allocation:

- `generated by default as identity`

Do this through a Drizzle migration.

This is the best-practice fit for a mostly auto-generated primary key that occasionally needs an explicitly reserved value.

## Apple Client Design

### Reservation Pool

Apple stores a tiny durable local pool of reserved chat IDs.

Pool policy:

- Refill when below `1`
- Refill back up to `3`

Reservations persist across restart/backgrounding.

Apple drops expired reservations from local storage during load and refresh.

Apple never creates local chats from the pool speculatively. A reserved ID is consumed only on explicit user create.

### Empty Pool Behavior

If the user creates a thread and the local pool is empty:

1. Show a spinner in the create flow.
2. Await `reserveChatIds(1)`.
3. Once returned, continue with the normal optimistic create path.

Do not add a separate direct-create fallback in this version.

### Local Optimistic Thread Creation

On create:

1. Consume reserved ID.
2. Insert optimistic local `Chat(id: reservedChatId)`.
3. Insert optimistic local `Dialog(chatId: reservedChatId, peerThreadId: reservedChatId)`.
4. Mark the chat as pending server creation.
5. Navigate immediately to the thread.
6. Enqueue `CreateChatTransaction(reservedChatId: reservedChatId, ...)`.

The optimistic local chat uses the final future server identity. No ID remap is needed later.

### Local Pending Chat Marker

Add a minimal local create-state marker on `chat`.

Recommended shape:

- `createState: text?`

Values:

- `nil`: confirmed / normal
- `"pending"`: local thread exists but server create not yet confirmed
- `"failed"`: create failed

This is intentionally small. It is enough for:

- spinner/loading state
- error state
- retry state

without introducing a larger local identity model.

### Failure and Retry

If create fails:

- set local `chat.createState = "failed"`

Retry behavior:

- consume a fresh reserved chat ID
- create a fresh optimistic local thread
- reopen the new thread

Do not mutate the failed optimistic thread in place.

## Realtime v2 Design

### Where This Logic Lives

Remote mutation dependency handling should live inside Realtime v2 transactions, not outside it.

Reason:

- Realtime v2 already persists and retries mutation transactions
- `CreateChatTransaction` and `SendMessageTransaction` already live there
- Later media-upload gating should reuse the same prerequisite mechanism for RPC execution

Uploads themselves remain outside Realtime v2. The dependency gating for RPC execution belongs inside Realtime v2.

### Blockers / Prerequisites

Add blocker support to the transaction system.

Initial blocker type:

- `chatCreated(chatId: Int64)`

Behavior:

- `CreateChatTransaction` for a reserved chat has no blockers
- `SendMessageTransaction` for a pending-created chat is enqueued immediately but blocked on `chatCreated(chatId)`
- When `CreateChatTransaction` succeeds, it publishes completion of `chatCreated(chatId)`
- The queue wakes and blocked sends for that chat become runnable

This should be modeled as a generic prerequisite mechanism so a future step can add:

- `uploadsFinished(localMessageId)`

for media send flows.

### Queue Execution Model

The queue should be able to hold blocked transactions durably.

At a high level:

- Persist transactions as today
- Persist blockers alongside transaction metadata
- Skip blocked transactions during dequeue/run
- Re-signal queue execution when a blocker is satisfied

The first supported dependency is chat creation confirmation.

## Data Flow

### Happy Path

1. Background refresh keeps Apple reservation pool warm.
2. User taps create thread.
3. Apple consumes reserved ID.
4. Apple writes optimistic local `Chat` and `Dialog`.
5. Apple marks chat pending.
6. Apple navigates immediately to the thread.
7. Apple enqueues `CreateChatTransaction`.
8. User sends message.
9. Apple writes optimistic local message immediately.
10. Apple enqueues `SendMessageTransaction`, blocked on `chatCreated(chatId)`.
11. `CreateChatTransaction` succeeds.
12. Apple clears pending create state on the chat.
13. Realtime v2 satisfies `chatCreated(chatId)`.
14. Blocked send transactions become runnable and send normally.

### Restart Behavior

After app restart:

- Reservation pool reloads from durable storage
- Expired reservations are dropped
- Pending `CreateChatTransaction` reloads from disk
- Pending blocked `SendMessageTransaction`s reload from disk
- Once transport is ready, Realtime v2 resumes create/send ordering naturally

## Expiration Policy

Use a long expiration window.

Principles:

- Expiration exists to clean up abandoned claims, not to enforce a tight online window
- The pool should remain fresh periodically and opportunistically
- Long-lived reservations are acceptable for this Apple-first design as long as the local pool is kept very small

The exact duration can be chosen during implementation, but it should be measured in days rather than minutes.

## Open Implementation Details

These were intentionally left for implementation design:

- Exact Drizzle migration for `chats.id`
- Exact `reserveChatIds` RPC schema
- Exact local reservation storage model on Apple
- Exact Realtime v2 blocker persistence and queue wake-up mechanics
- Exact UI for pending/failed thread create states

## Why This Approach

This is the narrowest design that solves the immediate UX problem cleanly:

- No route identity swap
- No local conversation identity refactor
- No temporary chat IDs
- Reuses existing Realtime v2 transaction model
- Builds a small prerequisite mechanism that can later be reused for media send gating

It intentionally optimizes for the current Apple thread-creation workflow without committing the system to a full local-first architecture.
