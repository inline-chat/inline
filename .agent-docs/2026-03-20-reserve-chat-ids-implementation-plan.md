# Reserve Chat IDs Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add Apple-first reserved chat IDs for immediate new-thread navigation, with pending chat state and Realtime v2 send-message blocking until chat creation is confirmed.

**Architecture:** The server allocates future `chats.id` values through `reserveChatIds` and lets `createChat` claim an optional `reservedChatId`. Apple keeps a tiny durable pool of reserved IDs, creates optimistic local chats using the final future ID, and uses Realtime v2 transaction blockers so `SendMessageTransaction` can queue behind `CreateChatTransaction` without changing route identity.

**Tech Stack:** Bun/TypeScript server, Postgres/Drizzle, protobuf RPC, Swift/GRDB on Apple, Realtime v2 transaction persistence and queueing.

## Notes For Execution

- Keep the reservation model minimal. Do not add token/type/scope columns unless implementation reveals a concrete correctness gap.
- Keep the pool policy fixed at: refill when below `1`, back up to `3`.
- Keep the first blocker API minimal. Only `chatCreated(chatId)` is required in this phase.
- Do not extend this plan to subthreads or full offline-first identity changes.
- For server create retries, preserve current validation behavior and keep claim logic transactionally tight.
