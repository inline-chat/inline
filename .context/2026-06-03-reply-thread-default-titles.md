# Reply Thread Default Titles Plan

## Goal

Move reply-thread fallback titles from iOS/macOS presentation code into backend `chat.title`, using `Re: <msg snippet>` by default. Add a chat-level flag so clients can distinguish a server-generated default title from a user/API-renamed title.

## High-Level Plan

- Backend owns reply-thread default titles: when a reply-thread is created without an explicit title, store `Re: <msg snippet>` in `chat.title`.
- Store title state explicitly: add an optional true-only `isUntitled`/`is_untitled` so clients know a thread has not been explicitly titled, even when `chat.title` contains generated display text.
- Keep rename behavior simple: any user/API title update clears the untitled marker; generated defaults set it to true.
- Simplify iOS/macOS title display: remove anchor-message title calculators and render `chat.title` directly, with only small legacy fallbacks.
- Preserve empty-thread cleanup: use the untitled marker instead of empty title to decide whether an empty reply thread can be auto-deleted.

## Current State

- Backend `chats.title` is nullable (`server/src/db/schema/chats.ts`) and `createSubthread` currently inserts `null` when no explicit title is supplied.
- `CreateChatInput.title` is currently a non-optional protobuf string; this change should make it optional so omitted title is the normal untitled-thread signal.
- `server/src/modules/subthreads.ts` already has `buildDefaultReplyThreadTitle(anchorMessage)`, which returns `Re: <collapsed 72-char snippet>` or `Re: Message`.
- `proto/core.proto` `Chat` has fields through `parent_message_id = 12` and no title-source flag.
- `UpdateChatInfo` only carries optional `title`/`emoji`, so title-source changes would not sync through update streams unless the update payload also carries the flag.
- Apple local `Chat` only stores `title`; cleanup uses empty title as the untitled signal in `DataManager.deleteThreadIfUntitledAndEmpty`.
- macOS fallback calculators are in `ReplyThreadTitleFallback`, plus sidebar/all-chats/title-model callers.
- iOS fallback calculator is `ReplyThreadToolbarContextLoader` and the chat title extension reads it for reply threads.

## Contract

Use an optional boolean named `untitled` on the protocol model, backed internally by `chats.is_untitled` / `Chat.isUntitled`.

Semantics:

- `untitled = true`: the thread has not been explicitly titled. `chat.title` may be `nil`/empty or may contain backend-generated display text like `Re: anchor`.
- field absent / `nil` / `undefined`: title is explicit/custom, legacy, or unknown. Treat this as not untitled.
- Apply this to thread chats only. Private chats/DMs should omit it even when their stored title is nil.

Only send the field when it is `true`; do not send `false`.

## Backend Plan

1. Add `isUntitled: boolean("is_untitled")` to `server/src/db/schema/chats.ts`.
2. Generate a Drizzle migration with `bun run db:generate <name>` from `server/`; review the generated SQL.
3. Backfill existing data in the migration:
   - leave existing rows `NULL`; before this change we cannot prove a title was generated by the backend.
   - only newly created untitled threads should get `true`.
4. Add `optional bool untitled = 13;` to `Chat` in `proto/core.proto`.
5. Change `CreateChatInput.title` to `optional string title` so omitted title is a real untitled signal.
   - Also treat trimmed empty title as untitled for compatibility with older/newer clients that still send an empty string.
6. Add `optional bool untitled = 4;` to `UpdateChatInfo` so realtime title updates can carry the flag when needed.
7. Regenerate protocol outputs (`bun run generate:proto`; include Swift regen if not covered, otherwise run `cd scripts && bun run proto:generate-swift`).
8. Update `encodeChat` to include `untitled: true` only when `chat.isUntitled === true`; otherwise omit it.
9. Update `createChat`:
   - normalize/trim title at the boundary;
   - if an explicit title exists, store it and leave `isUntitled` unset;
   - if a thread is created with no usable title, store `title: null` and set `isUntitled: true`.
10. Update `createSubthread`:
   - trim `input.title`;
   - if explicit title exists, store it and leave `isUntitled` unset;
   - if omitted and `parentMessageId` has an anchor, store `buildDefaultReplyThreadTitle(anchorMessage)` and set `isUntitled: true`;
   - if omitted without a reply anchor, store `title: null` and set `isUntitled: true`.
11. Update `updateChatInfo` / `updateThreadInfo`:
   - any successful user/API title update clears `isUntitled` to `NULL`;
   - only include `untitled: true` in update payloads if a backend flow explicitly regenerates a default title.
12. Audit any server paths that update/clone/move chats (`forwardMessages`, `moveThread`, test helpers) so the flag is preserved or set intentionally.

## Apple Plan

1. Add `isUntitled: Bool?` to `InlineKit` `Chat` and `ApiChat`, with a GRDB migration appended in `Database.swift`.
2. Update `Chat(from: InlineProtocol.Chat)` and all save/update paths to persist `proto.hasUntitled ? proto.untitled : nil`.
3. Update `UpdateChatInfo.apply` and `UpdateChatInfoTransaction.optimistic()` so any update carrying a title clears `chat.isUntitled` unless the payload explicitly has `untitled == true`; when a full `Chat` response is saved, preserve the server value.
4. Update `deleteThreadIfUntitledAndEmpty` to use `chat.isUntitled == true` (plus no messages) instead of empty title. This preserves auto-cleanup after backend starts writing default `Re: ...` titles.
5. Update create-thread transactions/UI to allow omitted/empty title to round-trip as an untitled thread if normal new-thread creation should support that flow.
6. Remove macOS fallback title calculators from active UI:
   - sidebar/all chats should use `chat.humanReadableTitle` / `chat.title` directly;
   - toolbar/window title should stop loading anchor text just to compute `Re: ...`;
   - keep parent-title loading only where navigation context still needs the parent peer/title.
7. Remove iOS reply-thread title placeholder calculation:
   - `ChatView+extensions` can read `chat.humanReadableTitle`;
   - `ReplyThreadToolbarContextLoader` should no longer fetch anchor message text for the child title, only parent context if still needed.
8. Keep a short fallback like `"Chat"`/`"New thread"` only for legacy rows from old servers or partially synced local state, not for normal reply-thread title generation.

## Tests

Backend focused tests:

- `messages.createSubthread.test.ts`: omitted title creates `Re: anchor`, stores `is_untitled = true`, and returns `untitled = true` in `Chat`.
- explicit subthread title stores title and leaves `is_untitled` absent/null.
- empty/whitespace explicit title falls back to generated title and `is_untitled = true`.
- no usable anchor text stores `Re: Message` with `is_untitled = true`.
- non-reply thread/subthread creation with no usable title stores `title = null` and `is_untitled = true`.
- regular `createChat` with omitted/empty title stores `title = null`, `is_untitled = true`, and encodes `untitled = true`.
- `messages.updateChatInfo.test.ts`: rename clears `is_untitled` and does not send `untitled: false`.

Apple focused tests:

- `Chat(from:)` maps optional `isUntitled`.
- GRDB migration preserves older rows with `nil`.
- `deleteThreadIfUntitledAndEmpty` deletes empty/no-message reply threads with generated titles only when `isUntitled == true`.
- sidebar/all-chat item tests no longer need DB anchor-text fallback.

## Rollout Risks

- Existing clients that do not know `untitled` will still display `chat.title`, so they benefit from backend defaults.
- New clients against old servers must tolerate missing `untitled` as not-untitled and keep a minimal display fallback.
- The main behavioral risk is accidental deletion of manually renamed empty threads; this is why rename/title update paths must clear the marker before removing title-placeholder heuristics.
